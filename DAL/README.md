# 🗄️ DAL — слой доступа к данным

Группа проектов **DAL** (Data Access Layer) образует промышленный, мультимандантный слой доступа к данным платформы. Реализована по принципам **гексагональной архитектуры** и **Clean Architecture**: бизнес-логика общается с БД исключительно через абстракции, а конкретные реализации (NHibernate, SQL Server, in-memory-кэш) подключаются как взаимозаменяемые адаптеры.

В состав группы входят три проекта:

| Проект | Назначение |
|---|---|
| **`DAL.Entities`** | Чистые POCO-сущности БД (`DbEntity`), атрибуты и расширения маппинга — без зависимостей от ORM. |
| **`DAL.SqlServer`** | Полноценная реализация persistence на **NHibernate + Microsoft SQL Server**: репозитории, шлюзы, сессии, транзакции, перехватчики, маппинг, HQL-генераторы, пользовательские типы. |
| **`DAL.Cache`** | Прозрачный декоратор-кэш над шлюзами на базе `IMemoryCache` с типобезопасными ключами и изоляцией ссылок через `Clone()`. |

---

## ✨ Ключевые возможности

- 🏛️ **Чистое разделение слоёв** — сущности, маппинг, репозитории, шлюзы и кэш разнесены по отдельным сборкам с минимальной связностью.
- 🔁 **Repository + Gateway pattern** — `Repository` отвечает за стандартный CRUD через NHibernate, `Gateway` инкапсулирует сложные запросы, JOIN-ы и хранимые процедуры.
- 🏢 **Мультимандантность из коробки** — каждый клиент (mandant) обслуживается на собственном шарде SQL Server, мастер-данные хранятся в отдельной БД `UNIVERS`.
- 🔐 **Контекст безопасности на уровне БД** — при открытии каждой сессии в SQL Server автоматически прокидываются `UserId`, `HostId`, `SessionId` через `sp_set_session_context`, что позволяет реализовывать аудит и Row-Level Security без изменения прикладного кода.
- ♻️ **Управление сессиями через `IDbSession`** — обёртка над `ISession` NHibernate с явным режимом открытия (`session.Open(mode)`), предсказуемым жизненным циклом и интеграцией с DI.
- 🚀 **Кэш-декораторы** — кэшированные шлюзы (`*CachedGateway`) прозрачно оборачивают обычные, поддерживают **sliding/absolute expiration** и **массовую инвалидацию** по предикату ключей (например, при logout пользователя).
- 🧬 **Безопасная отдача из кэша** — кэшированные объекты **клонируются** перед возвратом потребителю, что исключает побочные эффекты от мутаций на стороне сервиса.
- 📦 **Готовые подсистемы** — отдельные группы шлюзов и репозиториев под `Terminals`, `PassOffice`, `Deploy`, `Sessions` и общие справочники (классификаторы, хосты, пользователи, мандант).
- 🛡️ **Перехватчики и транзакции** — собственные `Interceptors` и `Transactions` обеспечивают логирование, контроль изменений и согласованность данных.

## ⚙️ Технологические особенности

- 🧩 **ORM:** **NHibernate** как основной маппер для боевого приложения; **EF Core** используется изолированно — только в проекте миграций (`Molcom.Migrations.Terminal`).
- 🗃️ **СУБД:** **Microsoft SQL Server** с использованием расширенных возможностей — `SESSION_CONTEXT`, табличные параметры (`DataTables`), хранимые процедуры, пользовательские типы (`UserTypes`).
- 🧠 **HQL-генераторы** — кастомные расширения языка запросов NHibernate под специфику предметной области.
- 🏗️ **Фабрики и декораторы** — `Factories` для конструирования сессий и соединений, `Decorators` для расширения поведения без изменения базовых реализаций (классический OCP).
- 💾 **Кэширование:** `IMemoryCacheManager` поверх `Microsoft.Extensions.Caching.Memory`, типобезопасные `CacheKey`-ы с переопределённым `Equals/GetHashCode`, конфигурация через `IOptions<CachePerTaskOptions>`.
- 💉 **DI-friendly дизайн** — все компоненты регистрируются как `Scoped` и дополнительно как `Lazy<T>` для отложенной инициализации тяжёлых зависимостей.
- 🧱 **POCO-сущности без протечек ORM** — `Molcom.DAL.Entities` не зависит ни от NHibernate, ни от EF, что позволяет переиспользовать модель в любых адаптерах persistence.
- 🌐 **C# 12 / .NET 9** — современный синтаксис: primary constructors, target-typed `new`, `record`-типы, `GlobalUsings`.
- ✅ **Проверяемость** — за счёт интерфейсной развязки (`IDbSession`, `I*DbGateway`, `I*CachedGateway`) слой легко покрывается интеграционными и модульными тестами без поднятия реальной БД.
